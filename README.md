# QA Portfolio: AI学習レコメンダー

## プロジェクト概要

本リポジトリは、中高生向けAI学習レコメンド機能を題材にした、AI SaaSプロダクトのQAポートフォリオである。PRDの受領からテスト計画・設計・実装・実行までのQAプロセスを一気通貫で設計し、**決定的なロジックと生成的なAI出力が混在するシステム**に対するテスト戦略を構造化して示すことを目的としている。

単なる機能テストの設計にとどまらず、AI出力の非決定性にどう対処するか、仕様の曖昧さをどう扱うか、他職種との責任分界をどう引くかといった、AI SaaS特有のQA判断を言語化した成果物である。

## ポートフォリオの意図

- AI出力のテスト戦略を設計できることを示す（決定的ロジックと生成AI出力の分離、テストレベルの設計）
- 仕様の曖昧さを発見し、仮定を明示してテスト設計を進められることを示す
- QAの守備範囲と他職種（PdM・Dev・SRE・セキュリティ）との棲み分けを理解していることを示す
- JSTQB準拠のテストプロセスに基づき、計画から設計までを一貫してドキュメント化できることを示す

## テスト対象システム（SUT）

生徒の学年・理解済み単元・苦手単元を入力として、次に学ぶべき単元をAIが推薦するWebアプリケーション。

アーキテクチャは以下の2層構造を採用している。

- **単元選定層（ルールベース）**: 学習指導要領に基づく単元依存関係を用いて、推薦すべき単元を決定的に選定する。
- **推薦理由生成層（生成AI）**: 選定された単元に対して、生成AIが推薦理由の文章を生成する。

この分離により、テスト戦略も2層に分かれる。単元選定の正しさはE2Eテストで決定的に検証し、推薦理由の品質はAI出力品質テストで確率的に評価する。

## ロール分け

本リポジトリは1人で作成しているが、実務のワークフローを再現するため、コミットメッセージのプレフィックスで各ロールの責務を明示している。

- `[PdM]` プロダクトマネージャー（企画・受け入れ条件定義）
- `[Dev]` 開発エンジニア（実装）
- `[QA]` QAエンジニア（テスト計画・設計・実行）

GitHub Issuesでは、QA-Dev間のコミュニケーション（バグ報告、設計方針の議論）も再現している。

## 主要ドキュメント

以下の順序で読むと、実務のワークフローに沿って理解できる。

1. **PRD（[`docs/prd.md`](./docs/prd.md)）**
   PdMが作成した企画ドキュメント。意図的に未決事項を残し、QAがフィードバックする前提で書かれている。

2. **テスト計画書（[`docs/test-plan.md`](./docs/test-plan.md)）**
   QAがPRDに基づき作成した戦略ドキュメント。PRDの未決事項に対する仮定、アーキテクチャ前提、テストレベルの分離方針、品質メトリクス、テスト終了条件を記述。

3. **テスト設計書（[`docs/test-design.md`](./docs/test-design.md)）**
   テスト計画に基づく具体的なテストケース一覧。AC・テスト観点・テストケースの3層トレーサビリティを確保。

4. **SUT（[`src/`](./src/)）**
   開発エンジニアが実装したレコメンダーアプリ。ルールベース + 生成AIの2層アーキテクチャ。

5. **QA-Devコミュニケーション（GitHub Issues）**
   [Issue #1](../../issues/1)（UI表示バグの発見と修正）、[Issue #2](../../issues/2)（UI構造改善の議論）。

## 主な取り組み

### JSTQB準拠のテストプロセス

要求分析（PRDレビュー）から、テスト計画、テスト設計、テスト実行、リリース判定までの一連のプロセスを、JSTQBシラバスの枠組みに沿って構造化している。同値分割・境界値分析などのテスト技法も適用。

### AI出力のテスト戦略の設計

生成AIの非決定的な出力に対して、「決定的ロジックと生成的出力を分離する」「品質メトリクスを定義し閾値で判定する」「実行頻度をコストと重要度で設計する」という3つの方針でテスト戦略を構築。

### 仕様の曖昧さへの対処

PRDの未決事項（推薦件数の可変性、推薦理由の文字数、一貫性要件など）に対して、仮定を明示した上でテスト設計を進め、PdMへの確認事項として計画書に記録する手法を採用。アジャイル開発における並走スプリントで QA が仕様確定を待たずに動き出す実務パターンを再現。

### 責任分界の明示

QA・PdM・Dev・セキュリティ・SRE など各職種の守備範囲を意識し、テスト計画書のスコープ定義で「何をテストしないか、なぜしないか」を明記。QAの越権を避ける設計判断を示している。
