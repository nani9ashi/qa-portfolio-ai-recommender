# テスト完了レポート: AI学習レコメンド機能 v1

## 1. エグゼクティブサマリー

本レポートは、AI学習レコメンド機能 v1 のテスト実施結果と、それに基づくリリース可否判定をまとめたものである。

テストは E2Eテスト（Playwright + TypeScript、18件）と AI出力品質テスト（Python + pytest、13件）の2層構造で設計・実施した。E2Eテストは全件グリーンを達成し、AI出力品質テストは Phase 1（ルールベース判定）で 76.92% だった合格率が Phase 2（LLM-as-a-judge 導入）で 92.31% となった。これは SUT（被テスト側のAI）の出力品質が向上したのではなく、ルールベース判定で誤判定となっていた妥当な出力を、LLM-as-a-judge への切り替えにより正しく合格判定できるようになった結果であり、判定設計側の成果である。テスト計画書で定義した品質メトリクスはすべて合格基準を満たしており、リリース可能な品質水準に到達したと判定する。

テストプロセスを通じて8件の Issue を発見・対応し、うち5件を Closed、3件を継続検討事項として Open のまま残した。Open の3件はいずれもユーザー価値の改善提案または v2 以降の検討事項であり、v1 のリリースを阻害するものではない。

特筆すべき成果として、Phase 1 で発見された3件の構造的課題（Issue #5/#6/#7）は Phase 2 の LLM-as-a-judge により実用上解消され、同時に Phase 1 では検出不可能だった新規課題（Issue #8）が判定厳格化を通じて発覚した。「ルールベース → LLM-as-a-judge」の段階的設計が双方向に機能することが定量的に裏付けられた点は、本プロジェクトのテスト戦略の有効性を示す重要な結果である。

> 詳細は以降の章で示す。

## 2. テストスコープと方針

### 2.1 テスト対象

本テストの対象は、中高生向け数学学習を支援する **AI学習レコメンド機能 v1** である。生徒の学年・理解済み単元・苦手単元の入力に対し、次に学ぶべき単元を1〜3件提案するWebアプリケーションであり、企画意図は [docs/prd.md](./prd.md) に定義される受け入れ条件 AC-1〜AC-5 として明文化されている。v1 では数学のみを対象とし、入力画面・結果画面のUI、ルールベースの単元選定エンジン、生成AIによる推薦理由生成の3要素を検証範囲とする。

### 2.2 テストレベル

決定的に振る舞う部分と確率的に振る舞う部分を分離し、2層のテストを並走させる方針を採用した ([docs/test-plan.md](./test-plan.md) §2 参照)。

| テストレベル | 対象 | 実装 | 性質 |
|---|---|---|---|
| **E2E テスト** | UI操作・画面遷移・ルールベース選定・APIエラー処理 | Playwright + TypeScript ([tests/e2e/](../tests/e2e/)) | 決定的・自動実行 |
| **AI出力品質テスト** | 生成AIが返す推薦理由の正確性・一貫性・安全性・品質 | Python + pytest ([tests/ai-quality/](../tests/ai-quality/)) | 確率的・手動実行 |

E2Eテストでは AI API を `page.route()` モックで遮断し、ルールベース部分の決定論的検証に集中する。AI出力品質テストでは実 Claude API を呼び出し、ゴールデンテストセット (GT-001〜005) を入力としたメトリクス評価を行う。

### 2.3 アーキテクチャ前提

決定的に振る舞うルールベース単元選定エンジンと、確率的に振る舞う生成 AI（Claude API）の 2 層構造を前提とする。詳細図と責務分離の意図は [test-plan.md §1.3](./test-plan.md#13-アーキテクチャ前提) を参照。両層は独立した責務として扱い、合否判定基準も別個に設定する。

### 2.4 テスト対象外

以下は本テスト計画のスコープ外とする ([docs/test-plan.md](./test-plan.md) §1.2)。

- **AI APIの内部ロジック・プロンプト設計**: エンジニアチームの設計判断であり、QAは出力結果のみを評価する
- **講師向け表示機能**: v1 スコープ外、v2 以降の検討事項
- **セキュリティテスト**: セキュリティチームの管轄。テスト実行中に懸念を発見した場合は別途報告
- **インフラ・デプロイテスト**: SREチームの管轄

### 2.5 PRDの未決事項に対する仮定

PRDには未決事項が複数含まれており、QAは確認待ちで作業を止めず、明示的な仮定を置いた上でテスト設計を進めた。主な仮定を以下に示す ([docs/test-plan.md](./test-plan.md) §1.4)。

- 推薦件数: 1〜3件の可変 (中1初期等で3件確保できないケースを許容)
- 推薦理由の文字数: 50〜150字 (意味のある2文程度〜冗長にならない長さ)
- 同一入力の一貫性: 推薦単元は完全一致、推薦理由の文章表現は揺れを許容
- AC-3 「優先度に反映」の解釈: 苦手関連単元が推薦リストに含まれていれば充足

### 2.6 想定体制

PdM・開発エンジニア・QA エンジニアの3者協働を前提とし、未決事項の確認、バグ報告、リリース判定の各タイミングで連携する想定で計画を立てた。本ポートフォリオでは1人が全役割を担うため、コミットメッセージのプレフィックス (`[PdM]` / `[Dev]` / `[QA]`) で役割を明示することで実務ワークフローを再現している。

> 上記スコープと方針に基づき、E2E テストおよび AI出力品質テストを実装・実行した結果を、次章でサマリする。

### 4.2 Phase 2 判定経路の集計

Phase 2 ベースラインにおける、各テストの最終判定がどの経路で確定したかを以下に集計する (`summary.json` の `by_judgment_path` より)。

| 経路 | 件数 | 該当テスト |
|---|---|---|
| `rule alone (passed)` | 6 | AI-C-001 / AI-C-002 / AI-Q-001 / AI-Q-003 / AI-S-001 / AI-S-003 |
| `rule + llm agreed (passed)` | 3 | AI-A-001 / AI-A-003 / AI-A-004 |
| `rule + llm rescued (passed: same intent across 10 runs despite set diff)` | 1 | AI-C-003 |
| `rule + llm rescued (passed: 6 reclassified as mentioned)` | 1 | AI-Q-002 |
| `rule + llm rescued (passed: 7 mentions reclassified as valid)` | 1 | AI-S-002 |
| `rule passed but llm rejected (final: failed)` | 1 | AI-A-002 |

ルール単独で判定済みが6件、ルール+LLM 合意が3件、LLM による合格への格上げが3件、LLM による不合格への格下げが1件である。LLM-as-a-judge が、合格への格上げと不合格への格下げの双方向に作用する設計が機能していることが確認できた。

### 5.2 開発フェーズで発見された課題（Issue #1, #2）

開発エンジニアによるルールベース実装の段階で、QAレビューを通じて発見された課題が2件ある。本ポートフォリオでは1人で全役割を担うが、コミットメッセージのプレフィックス（`[Dev]`/`[QA]`）と GitHub Issues 上の議論コメントによって、QA-Dev 間のコミュニケーションを実務ワークフローとして再現している。

Issue #1 は単元名と学年表記が重複して表示されるUI不具合として起票されたが、原因調査の過程で、データ層（`units.js`）と表示層（`renderUnitCheckboxes`）の両方で学年情報を付与する設計の二重化が判明した。最小修正としてはレンダリング側の付与を削除すれば足りるが、QAレビュー時に「より構造的な解決として、学年別の fieldset でグループ化するUI改善があり得る」との提案を行い、別 Issue（#2）として切り出した。

両者の修正は別コミットに分離し、Issue #1 の修正は単一の表記重複削除のみ、Issue #2 の修正は UI構造の再設計のみを含む構成とした。「1コミット1目的」を徹底することで、後から履歴を追う読み手が変更の意図を即座に把握できる構造を維持している。

### 5.3 E2Eテスト実装中に発見された課題（Issue #3, #4）

E2Eテスト設計を実装に落とし込む段階で、テスト設計書の期待結果と現実装の振る舞いに乖離があるケースが2件発見された。いずれも PRD の文言が複数解釈の余地を持つことに起因し、QA は v1 スコープとして現実装を受け入れる判断を行いつつ、改善提案として Issue 化することで意思決定を可視化した。

Issue #3 は学年切替時のチェックボックス表示が「累積表示」か「置換表示」かという仕様の曖昧さに起因する。PRD の「学年に応じて表示項目が変わる」という記述は両方の解釈を許容する。現実装は累積表示を採用しており、「下位学年を自動的に理解済み扱いにする代替案は製品コンセプトと矛盾する」という判断を加えて、累積表示を v1 の正として受け入れた。UX観点での煩雑さは Issue として記録し、v2 以降の改善検討事項に回した。

Issue #4 は中1全単元を理解済みとした際の挙動に関する。PRD の「推薦候補0件のメッセージまたは上位学年への案内」という記述から、現実装の0件メッセージは AC-4（エラーハンドリング）を満たすが、ユーザー価値の観点では上位学年案内のほうが望ましい。これも v1 では現実装を受け入れ、Issue として記録した。

QAとしては、仕様の曖昧さを理由にテスト実装を止めず、明示的な仮定とその根拠を文書化した上で進めるアプローチを取った。具体的には、テスト設計書 §3.1 と §3.2 に「現実装を v1 の正として採用、改善は Issue #3/#4 参照」のような注記を追加し、判断の経緯と将来の検討余地が後から追跡できる構造とした。

### 5.4 AI出力品質テスト Phase 1 で発見された課題（Issue #5, #6, #7）

Phase 1 のルールベース判定によるベースライン実行（2026-05-08）で3件の不合格が発生した。原因はいずれもルールベース判定の構造的限界に起因し、それぞれ異なる種類の課題を表していた。

具体的には以下の3パターンに整理できる。

- **Issue #5（AI-C-003、出力の揺らぎ）**: 同一入力を10回実行した際、推薦理由内で言及される単元が6種類のパターンに揺らいだ。temperature=0.3 で抑えても、生成テキストの細部は完全には決まらないという生成AIの特性に起因する。

- **Issue #6（AI-Q-002、文脈理解の限界）**: 初学者ケース（GT-003、入力単元が空）で、判定ロジックが「入力された学年名・苦手単元・理解済み単元のいずれかが推薦理由に含まれているか」をキーワードレベルでチェックする構造のため、文脈的に初学者向けの妥当な推薦理由を「言及なし」と判定した。

- **Issue #7（AI-S-002、過剰検出）**: 推薦単元の前提チェーンに含まれない単元名が言及されると「誤った前提関係」として検出する判定ロジックが、教育的に妥当な未来単元への言及（例: 中1の平面図形の推薦理由で高1の図形の性質に触れる）を誤って問題ありと判定した。

これら3件の共通点は、文字列レベルの厳密判定では「意味的な妥当性」を表現しきれない点である。出力の揺らぎ、文脈理解の限界、過剰検出と表れ方は異なるが、根本原因はいずれも「ルールベースは表面的なパターンマッチングに留まる」という設計上の制約に帰着する。

これが Phase 2 で LLM-as-a-judge を導入する直接的な根拠となった。3件はそれぞれ独立した Issue として GitHub Issues に記録し、対処方針として Phase 2 における意味的判定の追加を明記した上で、Phase 2 ベースライン実行で実用上の解消を確認している。

### 5.5 AI出力品質テスト Phase 2 で発見された課題（Issue #8）

Phase 2 のベースライン実行（2026-05-09）では、Phase 1 不合格3件すべてが LLM-as-a-judge による意味的判定で合格に格上げされた一方、Phase 1 で合格扱いだった AI-A-002 が Phase 2 で不合格に転じる現象が発生した。これは LLM-as-a-judge による判定基準の厳格化が想定通り作用したことを示す。

具体的には、AI-A-002（学年=中2、理解済み=比例、苦手=一次関数）の推薦結果に対し、LLM-as-a-judge は以下のように評価した。

> 「『正の数と負の数』の推薦理由は一次関数との接続として一定の妥当性があるが、『平面図形』は比例・反比例との関連付けが不自然で教育的根拠が薄い。『データの分布』に至っては比例・反比例や一次関数との繋がりが極めて希薄で、傾向線の言及も中2段階では不適切。全体的に生徒の苦手克服という文脈から逸れた単元が含まれており、推薦理由が後付けの印象が強い。」

ルールベース判定では「比例」「一次関数」のキーワード含有のみを見ていたため合格扱いとなっていたが、LLM-as-a-judge は推薦単元と入力情報の教育的関連付けの弱さを意味的に指摘した。これは「機械的判定では捕捉できない『後付けの推薦理由』」という新規課題の発見であり、Issue #8 としてバックエンドプロンプトの改善対象として記録した。

Issue #8 は v1 リリース時点での修正対象とはせず、v2 以降の改善検討事項として残した。判定が「合格相当の品質」と「やや後付けの説明」の境界線上にあり、ユーザー学習を妨げる致命的な誤りは含まないと判断したためである。

### 5.6 課題発見プロセスから得られた示唆

8件の Issue を発見・対応する過程で、テストフェーズと発見される課題の性質に明確な相関があることが観察された。本サブセクションでは、QAとして得られた3つの示唆をまとめる。

**(1) 段階的テスト設計が網羅性を担保する**

開発フェーズでは表面的な不具合（UI重複、構造改善）が見つかり、E2Eテストフェーズでは仕様解釈の曖昧さが顕在化し、AI品質テスト Phase 1 では機械的判定の限界が露呈し、Phase 2 では意味的判定が引き出す新たな課題が発覚した。各フェーズで発見される課題の性質が段階的に深化しており、これは「単一のテストレイヤーでは捕捉できない種類の課題が存在する」という事実の定量的裏付けである。AI 関連プロダクトでは、決定的な振る舞いと確率的な振る舞いを分離し、それぞれに適したテスト手法を組み合わせることが、網羅性の担保に直結する。

**(2) 「Bug?」運用が仕様の曖昧さを管理する**

仕様か不具合かが境界線上にある事象は、AI 関連プロダクトでは特に多い。本プロジェクトでは Issue タイトルプレフィックスに `[Bug?]` を採用し、確定前の議論を Issue 本文の「『Bug?』とした理由」セクションに集約した。これにより、PdM 不在のポートフォリオ環境でも「QA がどう判断したか」が後から追跡可能な構造を維持できた。実務でも、判定の境界線にある事象を機械的に Bug/Enhancement に振り分けるより、議論の場として Issue を活用するほうが、結果としてリリース判定の質を高めると考えられる。

**(3) メタ評価が AI 品質テストに不可欠である**

AI 出力を AI で評価する LLM-as-a-judge では、判定 AI 自身の判定が信頼できるかを検証する仕組み（メタ評価）が不可欠となる。本プロジェクトでは、判定 AI の出力する判定理由を JSONL 形式で完全保存し、後から目視レビュー可能な構造を組み込んだ。実際に Phase 2 の判定理由を確認した結果、判定 AI が単に合否を返すだけでなく、「『データの分布』に至っては比例・反比例や一次関数との繋がりが極めて希薄」のような具体的な根拠を提示しており、これが Issue #8 の発見に直結した。判定 AI の判定理由を読むこと自体が、表面的な合否判定を超えた洞察を生む。AI 品質テストの設計では、判定の自動化に加え、判定の説明可能性とトレーサビリティを最初から組み込むことが重要である。

> 以上の課題発見と対応を踏まえ、次章ではテスト戦略全体の評価を行う。

## 6. テスト戦略の評価

### 6.1 2層テスト構造の評価

E2Eテスト（決定的）と AI出力品質テスト（確率的）を分離する2層構造は、本プロジェクトで意図通り機能した。E2Eテストはバックエンドの起動も実 API キーも必要とせず、毎 push の CI 実行を低コストで継続できた。一方、AI出力品質テストは生成 AI の確率的な出力を許容する設計とし、コスト管理の観点から手動実行に絞ることでクレジット消費を抑えた。

両層を独立した責務として扱うことで、テスト失敗の原因切り分けも容易になった。E2Eテストが失敗すればルールベース選定または UI の問題、AI出力品質テストが失敗すれば生成 AI の出力品質の問題、と原因が明確に分離されるため、修正対象の特定にかかる時間が大幅に短縮される。

ただし、両層の境界がやや曖昧な領域も存在する。AC-5（レスポンスタイム10秒以内）は機能要件としては E2E テストで検証したが、実 API のレスポンスタイムは AI出力品質テストで計測すべき性質である。本プロジェクトではモック側に意図的な遅延を挿入することで E2E テストでの計測を実現したが、本番環境のパフォーマンス評価には別途の負荷試験が必要となる。

### 6.2 Phase 1 → Phase 2 の段階設計の評価

Phase 1 のルールベース判定で先にベースラインを取り、その後 Phase 2 で LLM-as-a-judge を選択的に追加するアプローチは、結果として極めて有効に機能した。仮に最初から LLM-as-a-judge のみで設計していた場合、判定の根拠がブラックボックス化し、判定 AI の妥当性検証が困難となっていた可能性が高い。

Phase 1 で機械的判定可能な部分（文字数、文体、ハルシネーション検出）を確立しておくことで、Phase 2 では「ルール単独で十分な部分」と「意味的判定が必要な部分」を切り分け、コストと判定精度のバランスを取ることができた。判定経路の集計（§4.2）にも示した通り、ルール単独で6件、ルール+LLM 合意が3件、LLM による合格への格上げが3件、LLM による不合格への格下げが1件と、各経路がそれぞれ機能した。

特に、Phase 2 で合格への格上げと不合格への格下げの双方向に判定が動いた点は、段階設計の正当性を最も強く裏付ける結果である。LLM-as-a-judge は単なる「ルールベースの限界を補う再判定の仕組み」ではなく、ルールベースが見逃す品質課題を炙り出す「品質ゲート」としても機能した。

### 6.3 想定通りに機能した点と想定外の点

事前に想定し、想定通りに機能した点を以下に挙げる。

- 機械的判定とメタ評価可能な意味判定の分離による、判定の透明性確保
- `workflow_dispatch` による手動実行限定設計でのコスト管理
- 判定経路の集計フィールド（`by_judgment_path`）による、テスト戦略の効果可視化
- Issue #5/#6/#7 の Phase 2 での実用的解消

一方、当初の想定を超えた発見もあった。

- Phase 2 実装時に LLM judge プロンプトの変数エスケープ問題が発生し、LLM judge が全件フォールバック動作する状態を初回実行で検出した。`by_judgment_path` に「rule + llm fallback」が並ぶ集計を見て即座に異常を察知できたのは、判定経路を記録する設計を最初から組み込んだことの直接的な恩恵である。これは「メトリクス設計の重要性」を示す具体例として記録に値する。
- Issue #8 のような「LLM-as-a-judge による判定厳格化を通じた課題発見」は、当初 Phase 2 の主目的として強く意識していなかった機能だが、結果として LLM-as-a-judge の最大の価値の一つとなった。

### 6.4 改善余地

本プロジェクトのテスト戦略は v1 として有効に機能したが、継続的改善の余地も明確に認識している。

- ゴールデンテストセット（GT-001〜005）はサンプル数が限定的であり、学年・単元の組み合わせを十分に網羅しているとは言えない。Phase 2 のベースライン実行コストが約 $0.20/回と十分小さいことを踏まえ、v2 ではサンプル数を倍増させる余地がある。
- 判定 AI の判定の妥当性を担保するメタ評価は、現状トレースの目視レビューに留まる。判定 AI の判定をさらに別の AI で評価する多段構造、または定期的な人間レビューの仕組みは、判定の信頼性をさらに高める方向の発展余地である。
- Phase 1 と Phase 2 のベースライン結果を継続的に比較する仕組みは、現状「合格率の推移を追跡」テーブルへの手動追記に依存している。リグレッション検出の自動化は、リリース後の継続運用で重要となる。

### 8.2 v2 以降への提言

本プロジェクトのテストおよび実装を通じて、v2 以降の機能拡張に向けた提言が3点ある。

**(1) ゴールデンテストセットの拡張と継続運用**

GT-001〜005 はカバレッジ確保の出発点としては機能したが、学年・単元の組み合わせは限定的である。v2 では中1から高3までの全学年を最低2件ずつ網羅し、全 GT 数を10〜15件に拡張することを推奨する。各 GT に対する `verified_by_human` フラグの運用は継続し、人間レビュー済みの期待値とAI推論で初期化された期待値を区別可能な状態を保つ。

**(2) 複数科目への展開時のテスト戦略**

v1 は数学のみを対象としたが、v2 で英語や理科に展開する場合、本プロジェクトのテスト戦略はそのまま再利用可能である。ただし科目固有のドメイン知識（例: 英語の品詞関係、理科の単元前提）は科目ごとに定義する必要があり、ゴールデンテストセットも科目別に作成する。判定 AI のプロンプトも科目固有の専門家ペルソナに調整することで、判定精度の維持が可能となる。

**(3) 講師向け表示機能の追加時の検討事項**

v2 で講師向け表示機能を追加する場合、新たに「個人情報・センシティブ情報の取り扱い」「講師の判断と AI 推薦の併存」というテスト観点が発生する。これらは現状の AI出力品質テストとは異なる種類のリスク（情報漏洩、講師の自律性侵害）を内包するため、専用のテストレイヤーの追加検討が必要となる。

### 8.3 テスト戦略の継続改善

v1 リリース後の継続運用を見据えて、テスト戦略自体の改善方向性を以下にまとめる。

- **リグレッション検出の自動化**: 現状、Phase 1 と Phase 2 のベースライン比較は手動で行っている。ベースライン結果を CI 実行時に自動比較し、合格率の悪化を検出する仕組みを導入することで、プロンプト変更や AI モデル更新時のリグレッションを早期発見できる。
- **判定 AI の精度測定**: LLM-as-a-judge 自体の判定精度をどう測定するかは AI評価の本質的な課題である。判定 AI の判定理由をサンプリングして人間がレビューする運用を定期化し、判定の妥当性を継続的に確認する体制を提案する。
- **判定経路の可視化ダッシュボード**: `by_judgment_path` の集計値を時系列で可視化することで、テスト戦略の効果（合格への格上げ件数・不合格への格下げ件数の推移）を運用観点で追跡可能となる。

### 8.4 ポートフォリオとしての学び

本プロジェクトを通じて、AI 関連プロダクトの QA に求められる視座について、実装前の想定とは異なる気づきがいくつか得られた。

第一に、AI の確率的な出力を扱う際は「合否を判定すること」より「判定の根拠を記録・可視化すること」のほうが重要である。判定の自動化は手段であって目的ではなく、後から人間がレビューできる構造を最初から組み込むことが、AI 品質テストの実効性を左右する。

第二に、ルールベース判定と意味判定を併用する段階設計は、コスト効率と判定精度の両面で優れた結果をもたらす。「すべてを LLM judge に任せる」アプローチより、「機械的に判定可能な部分はルール、意味的判定が必要な部分のみ LLM」という切り分けのほうが、運用上の透明性と継続改善のしやすさで勝る。

第三に、QA は「壊れているものを直す」だけでなく「より良い構造への提言を行う」役割を担う。Issue #1 から派生した #2、Phase 1 の課題から派生した Phase 2 の設計、Phase 2 で発覚した #8 のバックエンド改善提案など、テスト活動はプロダクトの改善ループそのものを駆動する。これは AI 関連プロダクトに限らない普遍的な QA の役割であるが、AI の確率的特性によってその重要性がより顕著に表れる領域でもある。
