# テスト計画書: AI学習レコメンド機能

## 1. テスト計画の概要

### 1.1 目的
本テスト計画書は、AI学習レコメンド機能（PRD参照）のリリース前品質保証を目的とする。PRDに定義された受け入れ条件（AC-1〜AC-5）の充足確認、およびAI生成出力の品質評価を行う。

### 1.2 スコープ

**テスト対象:**
- AI学習レコメンド機能 v1（数学のみ）
- 入力画面・結果画面のUI
- ルールベースの単元選定エンジン
- 生成AIによる推薦理由生成

**テスト対象外:**
- AI APIの内部ロジック・プロンプト設計（エンジニアチームの設計判断であり、QAは出力結果のみを検証する）
- 講師向け表示機能（v1スコープ外）
- セキュリティテスト（セキュリティチームの管轄。ただしテスト中に懸念を発見した場合は報告する）
- インフラ・デプロイ（SREチームの管轄）

### 1.3 アーキテクチャ前提

本テスト計画は以下のアーキテクチャを前提とする。

```
入力（学年・理解済み単元・苦手単元）
↓
ルールベースの単元選定エンジン
↓
推薦単元が確定
↓
推薦単元と入力情報を生成AIに渡す
↓
生成AIが推薦理由の文章を生成
↓
推薦単元 + 推薦理由を返す
```

このアーキテクチャにより、単元選定の正しさはルールベースエンジンの責務、推薦理由の品質は生成AIの出力評価の責務、と明確に分離される。

### 1.4 PRDの未決事項に対する仮定

テスト設計にあたり、PRDの未決事項について以下の仮定を置く。本仮定はPdMへの確認が完了次第、更新する。

| 未決事項 | 仮定 | 根拠 |
|---|---|---|
| 推薦件数の固定/可変 | 1〜3件の可変とする | PRDの記述との整合性。中1初期等で3件推薦できない場合があるため |
| 推薦理由の文字数 | 50〜150字 | 下限は意味のある説明に必要な最低限（2文程度）、上限は冗長にならない読み切れる長さ |
| 同一入力の一貫性 | 推薦単元は完全一致、推薦理由の文章は表現の揺れを許容 | AIの生成的出力の性質上、文章の完全一致は求めない |
| 講師向け表示 | v1スコープ外 | 数学のみに絞ったv1の焦点を保つため |
| AC-3「優先度に反映」の解釈 | 苦手関連単元が推薦リストに含まれていることをもって満たすとする | PRDの文言の範囲内で過不足なく対応するため |

### 1.5 想定体制と役割分担

本テスト計画は以下の体制を想定する。

| 役割 | 担当 | 主な責務 |
|---|---|---|
| PdM | 1名 | PRD作成、未決事項への回答、リリース可否判断 |
| 開発エンジニア | 1名以上 | アプリケーション実装、AI APIプロンプト設計、バグ修正 |
| QAエンジニア | 1名 | テスト計画・設計・実装・実行、バグ報告、品質メトリクスの計測 |

PdM・開発エンジニアとは、未決事項の確認、バグ報告、リリース判定の各タイミングで連携する。

## 2. テストレベルとアプローチ

本機能のテストは、以下の2つのレベルに分離して実施する。

### 2.1 E2Eテスト（Playwright + モック）

**目的:** UI・機能面の品質保証。AC-1, AC-2, AC-3, AC-4, AC-5の充足確認。

**アプローチ:** AI APIをモック（固定レスポンス）に差し替え、アプリケーション全体の振る舞いを検証する。モックを使用することで、テスト結果がAI APIの出力変動に影響されず、安定した反復実行が可能になる。CI/CDパイプラインに組み込み、毎コミット時に自動実行する。

単元選定はルールベースエンジンの責務であり決定的な処理であるため、E2Eテストで単元選定の正しさ（AC-2, AC-3）も検証可能である。

### 2.2 AI出力品質テスト（Python + 実API）

**目的:** 生成AIが生成する推薦理由の文章の品質評価。

**アプローチ:** 実際のAI APIに対して評価用の入力パターンを送信し、出力の品質を評価する。生成的出力に対しては正解が一意に定まらないため、品質メトリクスを定義し閾値で判定する。コストとAPI利用量を考慮し、リリース前およびプロンプト変更時に手動実行する。

**テストレベルを分離する理由:**
- E2Eテストは高速・低コスト・安定的に反復実行できる必要がある。AI APIの非決定的な出力がE2Eテストの安定性を損なうことを防ぐ。
- AI出力品質テストはAPIコストが発生し、毎コミット実行はコスト・時間・結果の不安定性の3点で現実的でない。実行頻度と評価方法をE2Eとは別に設計する。

## 4. テスト設計方針

具体的なテストケースはテスト設計書に委ね、本セクションでは設計の枠組みを示す。

### 4.1 ACとテストカテゴリのトレーサビリティ

| AC | 内容（概要） | 対応するテストカテゴリ |
|---|---|---|
| AC-1 | 基本的なレコメンド表示 | E2E正常系 |
| AC-2 | 前提条件の尊重 | E2E正常系（ロジック検証） |
| AC-3 | 苦手単元の優先 | E2E正常系（ロジック検証） |
| AC-4 | エラーハンドリング | E2E異常系・境界値 |
| AC-5 | レスポンス10秒以内 | E2E非機能テスト |
| — | 画面の振る舞い（AC外） | E2E UIテスト |
| — | 推薦理由の文章品質（AC外） | AI出力品質テスト |

### 4.2 E2Eテストの設計方針

**正常系:** PRDで定義された通常の入力に対して、ルールベースエンジンが正しい単元を選定し、画面に推薦が表示されることを検証する。

**異常系・境界値:** 入力未選択、全単元を理解済みにした場合（推薦候補0件）、全単元を苦手にした場合、API接続失敗時のエラー表示など、正しくない状況での振る舞いを検証する。

**UIテスト:** 学年変更時のチェックボックス表示切替、画面遷移など、機能の出力ではなく画面の振る舞いを検証する。

**非機能テスト:** AC-5に対応するレスポンスタイムの計測。

### 4.3 AI出力品質テストの設計方針

本テストの対象は生成AIが生成する推薦理由の文章のみ。単元選定の正しさはE2Eテストでカバーするため、本テストの対象には含めない。

**正確性テスト（ゴールデンテストセット）:** 事前に定義した入力パターンに対して、推薦理由が妥当であるかを検証。妥当性の判定基準は実務ではPdMまたはコンテンツチームと合意する想定。本ポートフォリオでは学習指導要領に基づき自前で定義する。

**一貫性テスト:** 同一入力を10回送信し、推薦単元と推薦理由の一貫性を計測。推薦単元は完全一致を期待、推薦理由の文章は表現の揺れを許容。

**安全性テスト:** 推薦理由にハルシネーション（存在しない単元名、誤った前提知識関係）が含まれないか、学習内容と無関係な情報が含まれないかを検証。

**推薦理由の品質テスト:** 推薦理由の文字数が想定範囲（50〜150字）内であるか、入力された生徒情報に言及しているか、文体が統一されているかを検証。

### 4.4 入力パターンの選定方法

入力の組み合わせは網羅できないため、同値分割・境界値分析を用いて代表ケースに絞る。具体的なケース選定はテスト設計書で行う。

### 4.5 テストケースの優先度定義

テスト設計書で各テストケースに優先度を付与する際の判定基準を以下に定める。

| 優先度 | 判定基準 | 例 |
|---|---|---|
| 高 | ユーザーへの影響が大きいケース。失敗が学習体験を損なう、またはプロダクトの信頼性を損なう。 | 理解済み単元が推薦に含まれる、API障害時にアプリがクラッシュする、推薦単元が空になる |
| 中 | 主要機能の正常系。失敗するとユーザーが本来の価値を享受できない。 | 通常の入力での推薦表示、学年変更時のチェックボックス切替、レスポンスタイム |
| 低 | UIの細部、稀にしか発生しない境界値、UX上の軽微な問題。 | 推薦理由の文字数が想定範囲をわずかに超える、文体の細かな統一性 |

## 5. 品質メトリクスと合否基準

| メトリクス | 計測方法 | 合格基準 | テスト層 |
|---|---|---|---|
| 理解済み単元の非推薦率 | E2Eテスト | 100% | E2E（ロジック検証） |
| 前提知識の尊重率 | E2Eテスト | 100% | E2E（ロジック検証） |
| 苦手関連単元の推薦含有率 | E2Eテスト | 100% | E2E（ロジック検証） |
| 推薦単元の一貫性 | E2Eテスト | 100% | E2E（ロジック検証） |
| レスポンスタイム | E2E非機能 | 10秒以内 | E2E |
| 推薦理由の文字数適合率 | AI品質テスト | 90%以上 | AI品質 |
| 推薦理由のハルシネーション検出率 | AI品質テスト | 5%未満 | AI品質 |
| 推薦理由が入力情報に言及している率 | AI品質テスト | 90%以上 | AI品質 |
| E2Eテスト通過率（優先度別） | CI/CD | 高100% / 中90% / 低80% | E2E |

## 6. バグの優先度・重大度の定義

テスト中に発見されたバグの分類基準を以下に定める。優先度は対応の緊急度、重大度は不具合の影響の大きさを示す。

### 6.1 重大度（Severity）

| 重大度 | 定義 | 例 |
|---|---|---|
| Critical | システム全体が機能しない、またはデータ破損が発生する | アプリが起動しない、推薦APIが常時失敗する |
| Major | 主要機能が利用できない、または誤った結果を返す | 理解済み単元が推薦に含まれる、推薦が表示されない |
| Minor | 機能は動作するが、一部に問題がある | 推薦理由の文字数が想定範囲を超える、文体が統一されていない |
| Trivial | UIの軽微な不整合、誤字脱字 | ボタンのラベルの表記揺れ |

### 6.2 優先度（Priority）

| 優先度 | 定義 |
|---|---|
| 高 | リリース前に必ず修正が必要。Critical・Majorはこのカテゴリに分類されることが多い。 |
| 中 | リリースまでに修正が望ましい。Minorのうち頻出するものはこのカテゴリ。 |
| 低 | 次回以降のリリースで修正を検討。Trivialや稀にしか発生しないMinorはこのカテゴリ。 |

重大度はバグそのものの性質、優先度はリリース判断と紐付く。重大度が高くてもプロダクトの優先順位次第で優先度が下がることがある（例：使われていない機能のCritical）。最終的な優先度判断はPdMと協議する。

---

## 7. テスト終了条件

以下を全て満たした時点で、テスト完了とみなす。

- 全テストケースが優先度別の閾値を満たしている（高100%、中90%、低80%）
- 品質メトリクス（セクション5）が全て合格基準を満たしている
- 優先度高のバグが0件
- 優先度中以下の未解決バグについて、PdMがリリース許容と判断している

## 8. リスクと対応方針

| リスク | 影響度 | 対応方針 |
|---|---|---|
| AIがハルシネーションを起こし、推薦理由に存在しない単元名や誤った前提知識関係が含まれる | 高 | 安全性テストで検出し、プロンプト修正をエンジニアにフィードバック |
| AI APIのレスポンスタイムが10秒を超える | 中 | E2Eテストの非機能テストで計測。継続的に超過する場合はPdMと要件を再協議 |
| AI APIの障害・レートリミットによりテスト実施が滞る | 中 | テスト実行のリトライ機構を組み込む。本番環境のフォールバック動作（AC-4）の検証も含める |
| PRDの未決事項に対する仮定がPdMの想定と異なり、テスト設計の手戻りが発生する | 中 | 未決事項を計画書に明示し、テスト計画策定後速やかにPdMに確認を取る |
| AI APIのテスト実行コストが想定を超える | 低 | AI出力品質テストの実行回数を制限。E2Eはモックで実施 |
